# loom V1 角色拆分

生效日期：2026-04-08

本拆分基于以下文档与当前仓库现状：

- `docs/loom-phase1-architecture-design.md`
- `docs/loom-springboot-backend-module-design.md`
- `docs/loom-java-package-structure.md`
- `docs/requirements-pool-spec.md`

## 角色与主责

| 角色 | 主责 | 对应需求 |
| --- | --- | --- |
| webAI 架构师 | 冻结领域模型、模块边界、接口契约、事件契约和阶段范围 | `ARC-001` ~ `ARC-005` |
| 前端开发 | 把现有三栏壳层接入真实数据、流式输出、Context、Trace、Settings | `FE-001` ~ `FE-006` |
| 后端开发 | 从 template 基础服务迁移到 loom Phase 1 模块化单体，并补齐 API / SSE / 数据模型 | `BE-001` ~ `BE-007` |
| 经理 | 维护排期、依赖、变更控制、验收节奏和风险台账 | `PM-001` ~ `PM-005` |
| 测试 | 建立 Phase 1 测试计划、联调检查单、回归矩阵和发布验收清单 | `QA-001` ~ `QA-005` |

## 当前分工结论

- 架构师先完成边界和合同冻结，再允许前后端并行。
- 前端当前重点不是重做 UI，而是把已有壳层接入真实项目、会话、Trace、Context 数据。
- 后端当前重点不是继续扩展 template 信息面，而是尽快建立 loom 的项目、会话、流式与上下文基础模块。
- 经理要把“旧设计已清空、新设计生效”的事实落实到排期和变更控制里。
- 测试要尽快补齐新设计下的测试计划，避免只验证壳层视觉而不验证真实链路。

## 协作边界

- 前后端并行的前提是 `ARC-002` 已冻结。
- `BE-003` 与 `FE-003` 必须联动交付，不能单边先做成私有协议。
- `QA-001` 必须在 `BE-002` / `FE-002` 开始联调前落地。
- 经理负责在每轮迭代回看本拆分与需求池是否仍与设计一致。
