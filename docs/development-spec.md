# loom 开发规范

## 1. 目标

本规范用于约束 loom 项目的日常开发、联调、测试、发布前协作方式，确保团队在保持速度的同时，不牺牲一致性、可维护性和可追溯性。

本规范默认以以下文档为上游依据：

- [loom V1 设计方案](./loom-v1-design-spec.md)
- [loom 需求池规范](./requirements-pool-spec.md)

## 2. 核心原则

- 设计先行：实现必须对齐已确认设计，不允许边写边重新定义核心边界
- 契约先行：前后端并行开发以接口契约、数据结构和状态定义为边界
- 文档强维护：代码、文档、测试必须一起交付
- 小步提交：每次改动都应可理解、可评审、可回退
- 可验证：所有主流程必须有明确验证方式

## 3. 角色分工

### 3.1 架构师

负责：

- 领域模型、模块边界、状态流转、关键技术方案
- 前后端契约基线
- 配置作用域、Memory、Trace、Run 等核心模型定义
- 关键设计文档维护

### 3.2 前端开发

负责：

- 页面结构、组件实现、交互状态、空态/错态/加载态
- 与后端契约对齐
- UI 文档、页面说明、交互变更说明维护

### 3.3 后端开发

负责：

- 领域服务、接口、数据模型、异步流程、配置规则
- 与前端、测试对齐契约
- API、数据结构、运行说明维护

### 3.4 测试

负责：

- 测试计划、测试用例、回归范围、风险反馈
- 联调验证与发布前验收
- 测试结果文档维护

### 3.5 经理

负责：

- 排期、依赖管理、需求变更控制
- 决策对齐与节奏管理
- 上线窗口、验收节奏、风险升级

## 4. 标准开发流程

### 4.1 需求就绪

开发开始前，需求必须已达到 `Ready` 状态，并满足：

- 上游文档存在且已确认
- 角色拆分完成
- 验收标准明确
- 文档更新清单明确

### 4.2 方案确认

以下改动必须先完成方案确认，再进入实现：

- 数据模型变化
- API / Contract 变化
- 页面结构变化
- Trace / Run / Memory / Capability 边界变化
- OpenClaw / MCP 接入方式变化

### 4.3 契约冻结

前后端并行前，至少冻结以下内容：

- 接口路径、方法、字段
- 错误码或错误态口径
- 关键状态枚举
- 时间字段、分页字段、排序字段口径
- 兼容策略

未冻结契约时，不允许以“先写着”作为默认做法。

### 4.4 开发执行

开发阶段要求：

- 任务必须关联需求 ID
- 每次提交围绕单一意图
- 代码变更同步补文档
- 每日回填状态和阻塞项

### 4.5 联调

进入联调前，前后端必须完成：

- 契约自检
- 关键路径冒烟
- 必要的 mock 或测试数据准备

联调结果必须写回需求条目或测试记录。

### 4.6 测试与验收

测试通过前，不得宣称“功能完成”。

至少应覆盖：

- 主流程
- 关键异常路径
- 配置变体
- 回归影响面

### 4.7 发布准备

发布前必须确认：

- 需求状态为 `Accepted`
- 文档已更新
- 测试结论已回填
- 风险和回退已确认

## 5. 文档强维护机制

### 5.1 基本原则

所有开发活动必须强维护文档，执行标准如下：

- 设计变更先改文档，再改代码，或至少同一 PR 一起提交
- 实现变更必须同步更新受影响文档
- 测试结论必须留下可回看记录
- 上线和回退方式变化必须同步更新运行文档

### 5.2 文档门禁

以下任一项不满足，PR 不得合并：

- 未关联需求 ID
- 未说明文档影响
- 影响行为但未更新文档
- 影响接口但未更新契约说明
- 影响页面但未更新交互说明或截图
- 影响发布但未更新运行文档

### 5.3 推荐文档更新映射

| 改动类型 | 至少同步更新 |
| --- | --- |
| 页面布局、组件交互变化 | 页面说明、线框说明、设计说明 |
| 接口字段、状态机变化 | API / Contract 文档、架构文档 |
| 数据模型变化 | 数据模型说明、迁移说明 |
| Memory / Trace / Run 机制变化 | 架构说明、测试重点说明 |
| 配置项变化 | 配置文档、`.env.example`、部署文档 |
| 发布流程变化 | 发布手册、回滚手册 |

## 6. 分支与提交要求

- 所有开发在独立分支完成
- 分支名应体现需求意图，例如 `feature/conversation-workspace`、`fix/run-step-status`
- Commit 信息应能说明变更目的
- 不允许在未验证情况下把大批无关改动混入同一分支

## 7. PR 要求

每个 PR 至少应包含：

- 需求 ID
- 变更目的
- 影响范围
- 测试方式与结果
- 文档更新清单
- 风险与回退说明

如果涉及 UI，必须附截图或录屏说明。

如果涉及 API / Contract，必须附字段或示例变化说明。

## 8. 评审规则

评审重点依次为：

1. 是否符合需求与设计
2. 是否破坏系统边界
3. 是否引入回归风险
4. 是否同步更新文档
5. 是否具备足够测试证据

以下情况必须退回修改：

- 文档缺失
- 契约未对齐
- 测试证据不足
- 隐含变更未披露
- 风险未说明

## 9. 测试规范

### 9.1 基础要求

- 新功能必须补对应测试
- 修复缺陷必须补回归用例
- 主链路必须有冒烟验证
- 高风险变更必须扩大回归范围

### 9.2 推荐覆盖层次

- 单元测试：逻辑、状态转换、工具函数
- 集成测试：接口、持久化、事件流
- 端到端测试：关键用户路径
- 回归测试：受影响模块和上下游链路

### 9.3 测试输出

测试至少要沉淀：

- 测试范围
- 执行环境
- 执行结果
- 已知问题
- 是否允许发布

## 10. 变更控制

开发中如需变更已确认需求，必须区分：

- 小变更：不影响范围和验收标准，可在原需求下补充记录
- 中变更：影响某角色任务范围，需更新需求条目和文档
- 大变更：影响阶段目标、核心边界或发布时间，需重新评审

禁止通过口头同步替代正式变更记录。

## 11. Definition of Done

任一需求只有在以下条件全部满足后，才算完成：

- 功能已实现
- 自测通过
- 测试通过或已获得明确豁免
- 文档已更新
- 风险与回退已确认
- 需求状态已回填
- 相关角色已完成验收或签字确认

其中“文档已更新”是硬条件，不可省略。
