# Loom UI 设计原则

本文档定义 Loom 当前与未来 UI 迭代的统一判断标准。它不是一次性说明，而是一份持续演进的产品资产。每次重要 UI 调整都必须同步更新本文档与基线文档，避免实现和原则再次脱节。

## 1. 先统一壳层，再细化页面

- 标题栏、左栏、主内容区、右栏、弹层和设置页必须属于同一套产品语言。
- 新页面默认继承现有 shell token、圆角、阴影、按钮与表面关系，不允许另起一套“后台风”或“活动页风”。
- 页面个性应该来自信息结构和内容，不应该来自私有配色体系。

## 2. 视觉活力来自结构，不来自花哨装饰

- Loom 的风格目标是“明显品牌感，但克制不花哨”。
- 活力主要通过四件事建立：表面层级、焦点色、排版节奏、微动效。
- 高饱和大面积背景、强渐变铺满页面、过多彩色按钮都属于反例。
- 品牌色只用于品牌标识、焦点态、主操作和关键活动状态。

## 3. 信息密度优先于标签堆叠

- 任何不能帮助用户做决策的信息都应降级或移除。
- 会话卡片不展示“助手”“系统”这类无新增价值标签。
- 同一区域不能重复展示同一信息，例如思考耗时和回复耗时不能分别用两套控件重复出现。
- 默认视图优先呈现结果，次级信息通过弱提示、折叠或侧栏展开。

## 4. 流式体验必须连续

- 发送、等待、思考、正文应尽量在同一个回复位置连续过渡。
- 不允许先闪现一个 loading 卡片，再切换成另一张正式消息卡片。
- 思考过程默认弱展示，只作为次级元信息提示；用户主动查看时再在右侧栏展开完整内容。
- 动态状态要低打扰，但必须让用户感知“系统正在工作”。

## 5. 导航、项目、会话是同宗组件

- 左侧导航项、项目项、会话项本质上都是可切换列表行。
- 它们必须共享行高、圆角、hover/active 节奏和交互反馈。
- 层级关系通过缩进、图标和字重表达，而不是依赖额外装饰块。
- 设置入口属于系统入口，不应长期呈现选中态强调。

## 6. 输入框是工作台的一部分

- 输入框不是独立小组件，而是会话工作台的核心交互区。
- 它与主面板必须共享边框、表面、圆角、阴影和焦点体系。
- 需要明确的空态、聚焦态、提交中、错误态与长文本态。
- 辅助提示文案必须弱化，主操作只有一个。

## 7. 设置页遵循“列表常驻，表单按需出现”

- 用户进入设置页首先应该看到可操作对象，而不是大段说明文字。
- 模型列表常驻，新增与编辑通过弹层或抽屉进入。
- 页面级主操作只有“新增模型”；卡片级主操作只有“切换到此模型”。
- 测试、编辑、关闭等行为都属于次级操作，应使用中性按钮。
- 高级配置默认折叠，只在用户明确需要时展开。

## 8. 主题系统必须是产品级能力

- 主题偏好支持 `system / dark / light` 三态。
- 主题切换入口放在全局标题栏，不能埋进设置页深层。
- 明暗主题都必须定义完整语义 token：背景、表面、浮层、文本、边框、品牌色、状态色、滚动条、输入态、焦点态。
- 亮色与暗色都要保持同一品牌气质，而不是两套不同产品。

## 9. token 是唯一颜色与间距来源

- 页面不能继续扩散私有颜色和私有按钮风格。
- 新增状态色时，必须同时考虑默认、hover、active、disabled 以及亮暗主题表现。
- 新组件优先复用语义 token，只有在全局可复用时才允许新增 token。

## 10. 设计原则必须持续更新

- UI 设计不是静态冻结的。
- 每次重要迭代都应记录新的判断依据、验证结果和适用范围。
- 如果实现与原则冲突，优先修正文档或修正实现，不能让两者长期分离。

## 参考依据

- Atlassian Design System：强调产品级按钮层级、信息密度和一致性。
- Material Design 3：强调语义 token、主题系统和层级清晰。
- IBM Carbon：强调复杂产品场景下的结构一致性与可扩展性。
- OpenAI / Anthropic 公开产品与官方文档：为流式反馈、思考弱提示和模型切换提供产品参考。
